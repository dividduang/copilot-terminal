# 输入法候选框跳动问题分析与解决方案

## 问题描述

在终端中使用中文输入法时，输入法候选框位置不稳定，会随着终端内容刷新而跳动。

## 根本原因分析

### 1. 问题的核心时序

从调试日志可以看出问题的发生过程：

```
[IME-FIX] 组合开始 (compositionstart)
[IME-FIX] 捕获冻结锚点 {left: '0px', top: '231px', ...}
[TerminalPane] 刷新输出到终端 {length: 49, ...}
[IME-FIX] updateCompositionElements 调用 #5 {isComposing: true, beforeLeft: '0px', beforeTop: '231px', ...}
[IME-FIX] updateCompositionElements 原始方法执行后 {afterLeft: '0px', afterTop: '252px', changed: true}
[IME-FIX] 应用冻结锚点完成 {finalLeft: '0px', finalTop: '231px'}
```

**关键发现**：
- 冻结锚点在 231px
- 终端输出后，xterm.js 将位置更新到 252px（增加了 21px，正好是一行高度）
- 我们的代码将位置恢复到 231px

### 2. 为什么候选框仍然跳动？

**根本原因：时间差导致的视觉闪烁**

旧的修复方案存在一个致命缺陷：

1. **终端输出触发 DOM 更新** → xterm.js 的 `terminal.write()` 会导致：
   - 终端内容重绘
   - 光标位置可能变化
   - `updateCompositionElements` 被调用

2. **xterm.js 更新 textarea 位置** → 在 `originalUpdate.call(this, dontRecurse)` 执行时：
   - xterm.js 根据**当前光标位置**计算 textarea 应该在哪里
   - 如果终端输出导致内容滚动或光标移动，计算出的位置就会变化
   - textarea 和 compositionView 被移动到新位置（252px）

3. **我们的代码恢复位置** → 在 `applyFrozenAnchor` 执行时：
   - 将位置恢复到冻结的 231px

4. **输入法候选框已经跟随 textarea 移动了** → **问题的关键**：
   - 输入法候选框的位置是由**操作系统**根据 textarea 的位置实时计算的
   - 在步骤 2 中，textarea 被移动到 252px 时，输入法候选框**立即**跟随移动
   - 在步骤 3 中，textarea 被恢复到 231px 时，输入法候选框**再次**跟随移动
   - 这个过程虽然很快（在一帧内），但**仍然可见**，导致候选框闪烁/跳动

### 3. 为什么 MutationObserver 也无效？

MutationObserver 的问题在于：
- 它是**异步**的，在微任务队列中执行
- 当它检测到变化并恢复位置时，输入法候选框已经跟随 textarea 移动过一次了
- 即使恢复速度很快，用户仍然能看到候选框的位置变化

### 4. 问题的本质

**旧方案的问题**：我们允许 xterm.js 的原始方法执行，然后再恢复位置。这导致了：
- textarea 位置先变化（231px → 252px）
- 输入法候选框跟随移动
- 我们再恢复位置（252px → 231px）
- 输入法候选框再次跟随移动
- 用户看到候选框跳动

**正确的做法**：在输入法组合期间，**完全阻止** xterm.js 更新 textarea 位置，避免任何位置变化。

## 解决方案

### 核心思路

在输入法组合期间，阻止 xterm.js 的 `updateCompositionElements` 调用原始方法，直接应用冻结的锚点。

### 实现细节

修改 `src/renderer/utils/terminalImeFix.ts` 中的 `patchedUpdateCompositionElements` 方法：

```typescript
compositionHelper.updateCompositionElements = function patchedUpdateCompositionElements(dontRecurse?: boolean): void {
  const isComposing = compositionHelper.isComposing;

  // 如果不在组合状态，正常执行原始方法
  if (!isComposing) {
    originalUpdate.call(this, dontRecurse);
    return;
  }

  // 在组合状态下，首次调用时捕获冻结锚点
  if (!frozenAnchor) {
    // 先执行一次原始方法，获取正确的初始位置
    originalUpdate.call(this, dontRecurse);
    frozenAnchor = readAnchorFromElements(textarea, compositionView)
      ?? (core ? calculateCursorAnchor(terminal, core) : null);
  } else {
    // 已有冻结锚点，直接应用，不执行原始方法
    // 这样可以避免 xterm.js 根据新的光标位置更新 textarea，导致候选框跟随移动
  }

  // 应用冻结锚点
  isApplyingFrozenAnchor = true;
  applyFrozenAnchor(textarea, compositionView, frozenAnchor, true);
  isApplyingFrozenAnchor = false;
};
```

### 关键改进

1. **首次调用**：执行原始方法，获取正确的初始位置并冻结
2. **后续调用**：**跳过原始方法**，直接应用冻结锚点
3. **结果**：textarea 位置在整个输入法组合期间保持不变，输入法候选框不会跳动

## 预期效果

修复后，输入法候选框应该：
1. 在输入法组合开始时，固定在初始位置
2. 在整个输入法组合期间，即使终端有输出，候选框位置也不会变化
3. 在输入法组合结束后，textarea 位置恢复到当前光标位置

## 技术总结

这个问题的关键在于理解：
- **输入法候选框的位置是由操作系统根据 textarea 的位置实时计算的**
- **任何 textarea 位置的变化都会导致候选框立即跟随移动**
- **即使变化很快（在一帧内），用户仍然能看到跳动**
- **正确的做法是完全阻止位置变化，而不是变化后再恢复**

这是一个典型的"预防胜于治疗"的案例：与其让问题发生后再修复，不如从源头上阻止问题发生。
