# 输入法候选框跳动问题 - 根本原因分析

## 问题现象

在终端中使用中文输入法时，输入法候选框会随着终端内容刷新而不断跳动，表现为"哪个字符发生变化，候选框就往往跑到哪里去"。

## 根本原因

通过深入分析 xterm.js 源码，发现了问题的完整链条：

### 1. onRender 事件触发机制

**位置：** `node_modules/@xterm/xterm/src/browser/CoreBrowserTerminal.ts:420`

```typescript
this._register(this.onRender(() => this._compositionHelper!.updateCompositionElements()));
```

**关键点：**
- 每次终端渲染（包括输出新内容、光标移动等）都会触发 `onRender` 事件
- `onRender` 事件会自动调用 `updateCompositionElements()`
- 这个调用是**无条件的**，即使在输入法组合期间也会触发

### 2. updateCompositionElements 的行为

**位置：** `node_modules/@xterm/xterm/src/browser/input/CompositionHelper.ts:224-262`

```typescript
public updateCompositionElements(dontRecurse?: boolean): void {
  if (!this._isComposing) {
    return;
  }

  if (this._bufferService.buffer.isCursorInViewport) {
    const cursorX = Math.min(this._bufferService.buffer.x, this._bufferService.cols - 1);
    const cellHeight = this._renderService.dimensions.css.cell.height;
    const cursorTop = this._bufferService.buffer.y * this._renderService.dimensions.css.cell.height;
    const cursorLeft = cursorX * this._renderService.dimensions.css.cell.width;

    // 关键：更新 compositionView 和 textarea 的位置
    this._compositionView.style.left = cursorLeft + 'px';
    this._compositionView.style.top = cursorTop + 'px';
    // ...
    this._textarea.style.left = cursorLeft + 'px';  // ← 这里！
    this._textarea.style.top = cursorTop + 'px';    // ← 这里！
    // ...
  }

  if (!dontRecurse) {
    setTimeout(() => this.updateCompositionElements(true), 0);
  }
}
```

**关键点：**
- 该方法会根据**当前光标位置**重新计算 textarea 的位置
- 如果终端有输出导致光标移动，textarea 位置就会跟着变化
- 输入法候选框锚定在 textarea 上，所以候选框也会跟着跳动

### 3. 问题链条

```
终端输出新内容
  ↓
光标位置变化
  ↓
触发 onRender 事件
  ↓
调用 updateCompositionElements()
  ↓
根据新的光标位置更新 textarea.style.left/top
  ↓
输入法候选框跟随 textarea 移动
  ↓
用户看到候选框跳动
```

## 为什么之前的修复方案无效

### 第一版方案的问题

我们 hook 了 `updateCompositionElements()` 方法，在组合期间跳过执行原始方法，直接应用冻结的锚点位置。

**理论上应该有效，但实际无效的原因：**

1. **时序问题：** 虽然我们跳过了原始方法的执行，但在首次调用时仍然执行了一次原始方法来捕获初始位置。如果在这之后立即有终端输出，可能会在我们应用冻结锚点之前就触发了位置更新。

2. **MutationObserver 的局限性：** MutationObserver 是异步的，它检测到样式变化并恢复位置之间有延迟，这个延迟足以让输入法候选框"闪一下"。

3. **可能存在其他更新路径：** 除了 `updateCompositionElements()`，可能还有其他代码路径在更新 textarea 位置（虽然从源码分析来看，这是唯一的路径）。

## 新的解决方案

### 核心策略

**在组合期间完全阻止 `updateCompositionElements()` 的执行（除了首次捕获锚点）**

### 实现细节

```typescript
compositionHelper.updateCompositionElements = function patchedUpdateCompositionElements(dontRecurse?: boolean): void {
  const isComposing = compositionHelper.isComposing;

  // 如果不在组合状态，正常执行原始方法
  if (!isComposing) {
    originalUpdate.call(this, dontRecurse);
    return;
  }

  // 在组合状态下，首次调用时捕获冻结锚点
  if (!frozenAnchor) {
    // 先执行一次原始方法，获取正确的初始位置
    originalUpdate.call(this, dontRecurse);
    frozenAnchor = readAnchorFromElements(textarea, compositionView)
      ?? calculateCursorAnchor(terminal, core);
    return;  // 首次调用后直接返回
  }

  // 已有冻结锚点，完全阻止执行原始方法
  // 不执行任何操作，直接返回
  // 这样可以阻止 xterm.js 根据新的光标位置更新 textarea
  return;
};
```

### 关键改进

1. **首次调用后不再应用锚点：** 首次调用原始方法后，我们捕获锚点但不再手动应用，因为原始方法已经设置了正确的位置。

2. **后续调用完全阻止：** 一旦有了冻结锚点，后续所有调用都直接返回，不执行任何操作。这样可以：
   - 阻止 xterm.js 更新 textarea 位置
   - 避免触发 MutationObserver
   - 消除任何可能的时序问题

3. **保留 MutationObserver 作为保险：** 虽然理论上不再需要，但保留 MutationObserver 可以作为最后一道防线，防止其他未知路径修改位置。

## 预期效果

- 输入法组合开始时，捕获当前 textarea 位置
- 组合期间，无论终端如何输出，textarea 位置保持不变
- 输入法候选框锚定在固定位置，不再跳动
- 组合结束后，恢复正常的位置更新机制
