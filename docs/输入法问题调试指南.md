# 输入法候选框闪动问题调试指南

## 问题描述

在终端中使用中文输入法时，输入法候选框位置不稳定，会随着终端内容刷新而跳动。

## 调试步骤

### 1. 启用调试日志

代码中已添加详细的调试日志，默认启用。如需关闭，修改 `src/renderer/utils/terminalImeFix.ts` 中的：

```typescript
const DEBUG_IME = false; // 改为 false 关闭日志
```

### 2. 重新编译并运行

```bash
npm run dev
```

### 3. 复现问题

1. 打开终端窗口
2. 切换到中文输入法
3. 开始输入中文（触发输入法候选框）
4. 同时观察终端是否有输出（例如运行一个持续输出的命令）

### 4. 观察控制台日志

打开 DevTools (Ctrl+Shift+I)，观察以下关键日志：

#### 输入法事件日志

```
[TerminalPane] compositionstart 事件触发
[TerminalPane] compositionupdate 事件触发 { data: "..." }
[TerminalPane] compositionend 事件触发 { data: "..." }
```

#### IME 修复逻辑日志

```
[IME-FIX] 组合开始 (compositionstart)
[IME-FIX] updateCompositionElements 调用 #1 { isComposing: true, ... }
[IME-FIX] 捕获冻结锚点 { left: "40px", top: "12px", ... }
[IME-FIX] 应用冻结锚点完成 { finalLeft: "40px", finalTop: "12px" }
[IME-FIX] MutationObserver 检测到 textarea 样式变化 #1 { ... }
[IME-FIX] 组合结束 (compositionend) { updateCallCount: 5, mutationCount: 3, ... }
```

#### 终端输出日志

```
[TerminalPane] 刷新输出到终端 { length: 100, preview: "..." }
```

### 5. 分析日志

#### 关键指标

1. **updateCallCount**: `updateCompositionElements` 在一次输入法组合期间被调用的次数
   - 正常情况：1-3 次
   - 异常情况：>10 次（说明终端输出频繁触发更新）

2. **mutationCount**: MutationObserver 检测到样式变化的次数
   - 如果这个数字很大，说明有其他代码在不断修改 textarea 位置

3. **位置变化**: 观察 `beforeLeft/afterLeft` 和 `frozenLeft/currentLeft`
   - 如果 `afterLeft` 与 `frozenLeft` 不一致，说明 xterm.js 在移动 textarea
   - 如果 MutationObserver 频繁恢复位置，说明存在竞态

#### 可能的问题模式

**模式 1：频繁的 updateCompositionElements 调用**
```
updateCompositionElements 调用 #1
updateCompositionElements 调用 #2
updateCompositionElements 调用 #3
...
updateCompositionElements 调用 #20
```
→ 说明终端输出过于频繁，每次输出都触发位置更新

**模式 2：MutationObserver 频繁恢复**
```
MutationObserver 检测到 textarea 样式变化 #1 { needRestore: true }
MutationObserver 恢复 textarea 位置
MutationObserver 检测到 textarea 样式变化 #2 { needRestore: true }
MutationObserver 恢复 textarea 位置
```
→ 说明存在其他代码路径在修改位置，与我们的恢复逻辑形成竞态

**模式 3：位置在两个值之间跳动**
```
捕获冻结锚点 { left: "40px", top: "12px" }
updateCompositionElements 原始方法执行后 { afterLeft: "180px", afterTop: "60px", changed: true }
应用冻结锚点完成 { finalLeft: "40px", finalTop: "12px" }
updateCompositionElements 原始方法执行后 { afterLeft: "180px", afterTop: "60px", changed: true }
```
→ 说明 xterm.js 不断将位置移动到新的光标位置，我们不断恢复

### 6. 根据日志调整方案

根据观察到的模式，可能需要：

1. **如果是模式 1**：考虑在输入法组合期间暂停终端输出渲染
2. **如果是模式 2**：需要找到其他修改位置的代码路径并 hook
3. **如果是模式 3**：可能需要更激进的方案，例如完全阻止 xterm.js 更新位置

## 解决方案（已实施）

### 根本原因

输入法候选框跳动的根本原因是：
- 在输入法组合期间，终端输出触发 xterm.js 更新 textarea 位置
- textarea 位置变化导致输入法候选框跟随移动
- 即使我们立即恢复位置，用户仍然能看到候选框的跳动

### 修复方案

在 `src/renderer/utils/terminalImeFix.ts` 中实施了以下修复：
1. **首次调用 `updateCompositionElements`**：执行原始方法，捕获并冻结初始位置
2. **后续调用**：跳过原始方法，直接应用冻结锚点，避免任何位置变化
3. **组合结束**：清除冻结锚点，恢复正常行为

### 验证方法

运行单元测试：
```bash
npm test -- terminalImeFix.test.ts
```

实际测试：
1. 运行持续输出的命令（如 `ping localhost`）
2. 使用中文输入法输入
3. 观察候选框是否稳定
