 问题描述说明

  在应用主页点击卡片进入终端后，终端界面存在输入体验异常问题。
  具体表现为：在终端中使用中文输入法时，输入法候选框的位置不稳定，不会固定在一个稳定位置显示，而是会随
  着终端内容刷新、字符变化而不断闪动和跳动。实际观感是“哪个字符发生变化，候选框就往往跑到哪里去”，导致
  候选框位置飘忽不定，输入体验较差。

  该问题在普通终端场景下会出现，在分屏后的终端窗格中也容易出现。
  相比之下，在同一台 Windows 设备、相同输入法、相同输入内容下，使用 Windows Terminal 时输入法候选框显
  示正常，没有闪动、乱跳或跟随字符变化漂移的问题。

  实际表现

  - 输入中文时，输入法候选框位置持续变化
  - 候选框会跟随终端字符变化区域跳动
  - 候选框显示过程伴随明显闪动
  - 在分屏终端中问题同样存在
  - 与 Windows Terminal 的稳定表现相比，当前应用内终端表现异常

  期望表现

  - 输入法候选框应稳定显示在正确位置
  - 候选框不应随着终端字符重绘而来回跳动
  - 输入过程中不应出现明显闪动
  - 单终端与分屏终端下都应保持一致、稳定的输入法体验

  影响

  - 中文输入时视觉干扰明显
  - 候选词选择体验变差
  - 连续输入时容易分散注意力
  - 在终端高频输出或局部重绘场景下问题更明显

  补充说明

  - 同机同输入法在 Windows Terminal 中表现正常
  - 说明系统输入法本身大概率没有问题
  - 问题更可能与当前应用终端的光标定位、输入法候选框锚点或终端重绘行为有关

## 解决方案

### 最终方案：在组合期间禁用 xterm.js 渲染循环

**实施时间**：2026-03-11

**核心思路**：
在 IME 组合期间（compositionstart → compositionend），完全禁用 xterm.js 的渲染循环，防止 Canvas 重绘导致候选框位置被抢夺。

**实现细节**：
1. Hook xterm.js 的 `_renderService._renderRows` 方法
2. 在组合期间，使该方法直接返回，不执行任何渲染
3. 延迟 PTY 输出的刷新，避免触发渲染
4. 组合结束后，恢复原始的渲染方法和输出刷新

**相关文件**：
- `src/renderer/utils/terminalImeFix.ts`：IME 修复核心逻辑
- `src/renderer/components/TerminalPane.tsx`：终端窗格组件，集成 IME 修复
- `docs/输入法候选框修复方案-禁用渲染.md`：详细技术文档
- `test-ime-animation.js`：测试脚本，模拟动画输出

**测试方法**：
1. 运行测试脚本：`node test-ime-animation.js`
2. 在动画运行时输入中文
3. 观察候选框是否稳定

### 之前尝试的方案（均失败）

1. **修改 textarea 的 style.left/top**：无效，因为候选框位置不是由 textarea 的 style 决定的
2. **暂停终端输出刷新**：无效，因为 Canvas 仍在重绘
3. **使用 MutationObserver 恢复位置**：无效，因为候选框位置不是由 DOM 属性决定的
