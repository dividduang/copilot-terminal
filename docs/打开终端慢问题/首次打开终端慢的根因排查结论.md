# 打开终端慢的根因排查结论

## 现象

- 主界面点击卡片进入终端，PowerShell 提示符秒开（~300ms）
- 点击拆分窗格，新窗格要等待约 3 秒才显示 `PowerShell 7.5.4\nPS D:\tmp>`

## 根因

### 直接原因

`TerminalPane` 的 `terminal.onData()` 回调中有 `if (!isActiveRef.current) return;` 检查，导致**非 active 窗格的 xterm.js 协议响应被丢弃**。

PowerShell 启动时会发送终端能力查询（如 `ESC[c` — Device Attributes 请求）。xterm.js 收到后通过 `onData` 回调生成协议响应（DA response），需要回传给 PTY。但新拆分的窗格不是 active 状态，响应被 `isActiveRef` 检查拦截，PowerShell 等不到 DA 响应，**超时等待约 3 秒**后才继续启动。

第一个窗格因为是 active 的，DA 响应正常回传，所以秒开。

### 修复

去掉 `onData` 回调中的 `isActiveRef` 检查，让协议响应无条件回传 PTY：

```typescript
// 修复前
const disposable = terminal.onData((data) => {
  if (!isActiveRef.current) return;  // ← 这行导致非 active 窗格的协议响应被丢弃
  if (window.electronAPI) {
    window.electronAPI.ptyWrite(windowId, pane.id, data);
  }
});

// 修复后
const disposable = terminal.onData((data) => {
  // Protocol replies (DA/DSR/window reports) also flow through onData and
  // must reach the PTY even when the pane is not the active one.
  if (window.electronAPI) {
    window.electronAPI.ptyWrite(windowId, pane.id, data);
  }
});
```

这个修改是安全的：用户键盘输入只有在 xterm 获得焦点时才会触发，而非 active 窗格不会获得键盘焦点，不会误发用户输入。

### 独立 node-pty 测试验证

使用独立脚本测试 `useConptyDll=true` vs `useConptyDll=false`：

```
useConptyDll=true:  PowerShell 版本信息在 +3281ms（慢）
useConptyDll=false: PowerShell 版本信息在 +292ms（快）
```

这一度误导排查方向，但实际上 `useConptyDll=true` 慢是因为独立测试脚本中 xterm.js 不存在，没有任何东西回复 DA 请求，PowerShell 同样在等待 DA 响应超时。应用中第一个窗格快是因为它是 active 的，DA 响应正常回传。

## 排除的假设

| 假设 | 排除原因 |
|------|----------|
| ConPTY DLL 预热 | 预热在启动时完成，后续 spawn 不会重复预热 |
| PowerShell 并发启动资源争抢 | 独立测试证明并发启动时间一致 |
| tmux 环境变量注入 | 禁用 tmux 后问题依旧 |
| pty-resize / SIGWINCH 干扰 | 添加防抖后问题依旧 |
| TerminalPane key 包含 pid 导致重建 | 去掉 pid 后问题依旧 |
| earlyDataBuffers 数据丢失 | 数据确实到达了 xterm，但 PowerShell 本身没输出 |
| PowerShell 启动本身慢 | 第一个窗格证明同样条件下可以秒开 |
